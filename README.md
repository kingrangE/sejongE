# 세종대 통합 정보 RAG 챗봇

세종대학교의 학사일정·비교과 프로그램·연구실(교수 연구분야) 정보는 서로 다른 사이트에
흩어져 있다. 학생 입장에서는 "지금 신청할 수 있는 비교과가 뭐가 있지?", "내 관심사에 맞는
연구실은 어디지?" 같은 질문 하나를 풀려고 사이트 서너 곳을 돌아다녀야 한다.

이 프로젝트는 그 흩어진 정보를 주기적으로 수집·정규화해 한 곳에 모으고, 자연어 질문에
**근거(출처)와 함께** 답하는 RAG 챗봇을 만드는 것을 목표로 한다.

```
"인공지능 쪽을 해보고 싶은데 어떤 연구실이 있어?"   → 관심사 기반 연구실 추천
"이번 주에 학사 일정 뭐 있어?"                      → 학사일정 기반 응답
"지금 신청 가능한 비교과 알려줘"                     → 신청기간·자격 필터 기반 응답
```

세 질문은 겉보기엔 비슷하지만 검색 성격이 다르다. 연구실 추천은 **의미 유사도** 문제이고,
나머지 둘은 "지금", "이번 주", "내 학년" 같은 **시점·조건 필터링** 문제다. 이 차이를
어떻게 다루느냐가 이 프로젝트 설계의 핵심이다.

---

## 시스템 구성

```
        수집(ETL)                 색인                검색/생성             서비스
 ┌────────────────────┐   ┌──────────────┐   ┌────────────────┐   ┌──────────────┐
 │ 비교과 / 학사일정 /  │   │ SQLite        │   │ Retriever      │   │ FastAPI      │
 │ 공지 / 학과·교수     │──▶│ (진실 원천)    │──▶│  (Vector→Hybrid)│──▶│  /chat (SSE) │
 │  httpx · Playwright │   │ Chroma        │   │ + 시간/자격     │   │ + 세션       │
 │  변경감지 · 멱등     │   │ (벡터 + 메타)  │   │   하드 필터     │   │              │
 └────────────────────┘   └──────────────┘   └───────┬────────┘   └──────┬───────┘
        ▲ APScheduler                                 │ tool use          │
        │ 주기 크롤                                    ▼                   ▼
                                              ┌────────────────┐   ┌──────────────┐
                                              │ Agent           │   │ Next.js/React│
                                              │ (Claude, 되묻기)│   │  채팅 UI      │
                                              └────────────────┘   └──────────────┘
```

데이터는 **수집 → 색인 → 검색 → 응답**의 단방향으로 흐른다. 각 단계를 독립 모듈로 분리해,
한 단계의 구현을 바꿔도(예: 벡터 검색 → 하이브리드 검색) 나머지가 영향을 받지 않도록 했다.

## 설계에서 고민한 것들

이 프로젝트는 "RAG = 임베딩 후 유사도 검색"이라는 단순 공식이 실제 도메인에서 깨지는
지점을 다루는 데 시간을 많이 썼다. 주요 의사결정과 근거는 다음과 같다.

### 1. 시점·자격은 유사도가 아니라 하드 필터로 분리했다
"지금 신청 가능한 비교과"를 벡터 유사도로 풀면, 임베딩은 "신청 가능"이라는 **현재 시점**
개념을 알지 못해 이미 마감된 프로그램도 그럴듯하게 끌어온다. 그래서 검색을 두 축으로 나눴다.

- **랭킹 축** — 의미 유사도(임베딩)
- **필터 축** — 신청기간·학년·전공 같은 조건은 메타데이터 `where` 절로 **먼저 잘라내고**,
  그 안에서만 유사도 랭킹을 한다.

이렇게 하면 "지금 신청 가능", "이번 주"는 LLM이 환각할 수 없는 결정적 필터가 되고,
유사도는 *유효한 후보들 안에서* 가장 적절한 걸 고르는 역할만 한다.

### 2. 날짜는 epoch-day 정수로, 상태는 질의 시점에 다시 계산한다
- 날짜 범위 겹침 질의("이번 주 일정")를 벡터DB 메타데이터 필터로 풀려면 정수 비교가 가장
  깔끔하다. 그래서 모든 날짜를 `epoch-day`(1970-01-01 기준 일수) 정수로 함께 저장한다.
- 비교과의 "접수중/마감" 상태는 **크롤 시점이 아니라 질의 시점**에 신청기간과 비교해
  다시 계산한다. 크롤 때 "접수중"이었어도 사용자가 물어보는 순간엔 마감일 수 있기 때문이다.
  크롤 당시의 스냅샷 상태를 신뢰하지 않는다.

### 3. 검색은 단순 Vector RAG부터, 하이브리드는 측정 후 도입한다
한국어는 형태소 특성상 정확한 명칭·코드 검색에서 BM25 같은 어휘 검색이 강하지만,
처음부터 하이브리드(임베딩 + BM25 + RRF)를 깔면 복잡도와 튜닝 비용이 커진다.
그래서 `Retriever` 인터페이스를 두고 **v1은 Vector 단독**으로 구현해 성능을 직접 측정한 뒤,
부족한 카테고리(정확명·코드 등)가 확인되면 `HybridRetriever`로 **교체만** 하도록 설계했다.
필터 축(위 1번)은 랭킹 방식과 무관하게 v1부터 항상 적용된다.

### 4. 크롤링은 1회성이 아니라 멱등·증분 ETL로 운영한다
정보가 계속 바뀌므로(특히 비교과 마감) 수집은 주기적으로 돌아야 하고, 매번 전체를 다시
임베딩하면 비용이 든다. 그래서:

- 문서마다 정규화 텍스트의 `content_hash`를 저장하고, 재실행 시
  **NEW / CHANGED / UNCHANGED**로 분류해 **바뀐 것만 다시 임베딩**한다.
- 같은 크롤을 두 번 돌리면 두 번째는 전부 UNCHANGED가 되어 임베딩 호출이 0이 된다(멱등).
- 소스에서 사라진 항목은 삭제하지 않고 `is_active=False`로 **소프트 삭제**해, "지난 행사"
  같은 과거 질의에도 답할 수 있게 이력을 남긴다.
- 원본 HTML을 그대로 캐시해, 파서를 고쳐도 **재크롤 없이 재파싱**할 수 있게 했다.

### 4-1. 실제 사이트 조사로 드러난 제약 — JS SPA 대응
대상 사이트를 직접 열어보며 수집 방식을 정했다.

| 사이트 | 확인된 방식 |
|---|---|
| 비교과(두드림) | 목록·상세·페이지네이션이 모두 **JS로 렌더링**되는 SPA. 정적 요청은 첫 묶음만 반환 → 전체 수집엔 Playwright 필요 |
| 학사일정 | 포털(`portal.sejong.ac.kr`)은 **로그인(SSO) 필요**. 대신 로그인이 필요 없는 **공개 학사일정 페이지**를 찾아 그쪽을 사용 |
| 학과·교수 | 학과 목록은 정적이나 **교수소개 탭이 JS 로딩**, 일부 학과는 자체 홈페이지에 교수진·연구실 정보 보유 → Playwright + 학과별 어댑터 |

정적 크롤이 막히는 지점을 미리 확인해, "정적(httpx) → 실패 시 Playwright"로 폴백하는
구조를 잡았다. 로그인이 걸린 학사일정은 인증을 뚫는 대신 **공개 대체 소스**를 찾는 쪽을
택했다(안정성·유지보수·정책 측면).

### 5. 콘텐츠 유형(글/표/이미지)을 형태에 맞게 처리한다
대학 공지·비교과 안내는 줄글만 있는 게 아니라 **표(신청기간·대상이 표로)** 나
**포스터 이미지(텍스트가 이미지 안에 박힌)** 형태가 흔하다. 그래서 수집 단계에서 콘텐츠를
유형별로 분기해, 표는 행 단위로 자연어화 + 필드 추출, 이미지는 비전 모델로 텍스트를
추출한 뒤 공통 형태로 합쳐 임베딩한다(이미지 추출은 content-hash로 캐시해 중복 호출 방지).

### 6. "잘 동작하는지"를 확인할 수 있게 만들었다
- **관찰 가능성**: 크롤 실행마다 건수·변경·삭제·오류를 `run_ledger` 테이블에 적재한다.
- **검수 도구**: `inspect` 명령으로 수집 결과를 사람이 검수하는 Markdown 리포트를 만든다
  (프로그램별 신청기간·상태·모집현황·원본 링크·임베딩 텍스트). 실제 사이트와 1:1 대조용.
- **평가**: 답은 정답이 정해진 경우가 많아, 골드셋을 만들어 직접 채점하는 방식으로
  검증한다. Vector → Hybrid 전환도 같은 골드셋을 좌우로 두고 비교 채점해 결정한다.

## 기술 스택

| 영역 | 선택 | 이유 |
|---|---|---|
| 임베딩 | OpenAI `text-embedding-3-large` | 한국어 품질, 차원 축소 옵션. 변경분만 호출해 비용 통제 |
| 벡터DB | Chroma | 무설치 임베디드, 메타데이터 필터 일급 지원. 추후 Qdrant 교체 가능하게 래핑 |
| LLM / 비전 | Claude (Opus 4.8) | 한국어, tool use(되묻기·검색 호출), 이미지 텍스트 추출 |
| 크롤 | httpx + BeautifulSoup / Playwright | 정적 우선, JS 사이트는 헤드리스 브라우저로 폴백 |
| 스케줄 | APScheduler | 순수 파이썬, 도메인별 주기 크롤 |
| 저장 | SQLite | 진실 원천. 변경감지·소프트삭제·실행 이력 |
| 백엔드 | FastAPI (SSE 스트리밍) | 챗 엔드포인트·세션 |
| 프론트 | Next.js + React | 채팅 UI·출처 카드·프로필 패널 |
| 검증 | pydantic v2 | 모든 외부 입력/문서를 스키마로 강제 |

실행 환경은 macOS / Linux를 기준으로 하며, OS 종속 코드를 피했다.

## 개발 로드맵

기능을 한 번에 다 만들기보다, **공유 인프라 → 가장 어려운 도메인 한 줄기 → 확장** 순으로
리스크를 앞당겨 해소하는 단계를 세웠다.

| 단계 | 내용 | 핵심 의도 |
|---|---|---|
| **Phase 0** | 도메인 모델·시간 유틸·저장소·인터페이스 | 이후 모든 단계가 의존하는 토대를 먼저 고정 |
| **Phase 1** | 비교과 ETL 한 줄기(수집→정규화→색인, 멱등) | 시간·자격·변경감지 등 **가장 까다로운 도메인**을 먼저 끝까지 관통 |
| **Phase 2** | Vector 검색 + 대화형 되묻기 | 비교과 Q&A 완성, 수동 채점으로 검색 품질 측정 |
| **Phase 3** | 학사일정·연구실 확장 | 같은 ETL·검색 골격 재사용(연구실은 AI융합대학부터) |
| **Phase 4** | FastAPI + Next.js 통합 UI | 라우터·도구를 묶어 end-to-end 서비스화 |
| **Phase 5** | 지속 운영 + (필요 시) 하이브리드 검색 | 주기 크롤·운영 안정화, 측정 기반 검색 고도화 |

비교과를 1순위로 둔 이유는, 시점 필터·자격 필터·되묻기·변경감지 같은 어려운 요소가 모두
모여 있어 여기서 패턴을 검증해두면 나머지 두 도메인은 재사용으로 풀리기 때문이다.

## 현재 진행 상황

**Phase 0 · 1 완료** (테스트 30개 통과).

동작하는 것:
- 비교과(두드림) 수집 → 정규화 → 멱등 적재까지의 ETL 한 줄기. 실제 사이트로 검증.
- SQLite 진실 원천(변경분류·소프트삭제·실행 이력), Chroma·OpenAI 어댑터.
- 시간/자격 하드 필터, 한국어 상대 날짜 해석("이번 주", "지금" 등).
- `crawl` / `inspect` CLI — 수집과 검수.

정직한 제약(다음 단계에서 해소):
- 두드림이 JS SPA라 현재 정적 수집은 첫 묶음(약 7~8건)만 가져온다. 전체 목록·상세 본문·
  자격(학년/전공) 정밀 추출은 Playwright 도입 후 채워진다(현재 자격은 "전체"로 가정).
- 학사일정·연구실 수집, 검색·에이전트·API·프론트엔드는 로드맵 단계.

자세한 실행/구조는 [`backend/README.md`](backend/README.md) 참고.

## 빠른 시작

```bash
cd backend
pip install -e .

# 1) 수집이 잘 되는지 검수 (API 키 불필요)
python -m sejong_rag.cli inspect --site bigyogwa
#  → data/inspect_bigyogwa.md 생성. 실제 사이트와 대조.

# 2) 실제 색인 적재 (OpenAI 임베딩 + Chroma; .env 키 필요)
python -m sejong_rag.cli crawl --site bigyogwa

# 테스트
pytest -q
```

> Windows cmd에서는 `pip install -e .` 후 동일하게 `python -m sejong_rag.cli ...`로 실행.

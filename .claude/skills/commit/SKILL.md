# commit

세 가지 규칙을 항상 지킨다.

1. **Conventional Commits 형식** — `type(scope): 요약`
2. **메시지는 한글** — 요약과 본문 모두 한국어로 작성
3. **기능 단위 커밋** — 한 커밋은 하나의 논리적 변경만. 작업 트리에 여러 기능이 섞여
   있으면 나눠서 여러 번 커밋한다.
4. **co-author금지** — 당신이 커밋했다는 것을 절대 밝히지 마시오.

## 절차

1. **변경 파악.** `git status`와 `git diff`(스테이징 전), `git diff --staged`로 무엇이
   바뀌었는지 전부 확인한다. 추측하지 말고 실제 diff를 읽는다.
2. **기능 단위로 묶기.** 변경을 논리적 단위로 그룹핑한다. 예: "검색 기능 추가"와
   "오타 수정"은 별개 커밋. 한 파일 안에 두 기능이 섞여 있으면 hunk 단위로 나눈다
   (`git add -p`).
3. **단위별 스테이징.** 해당 단위의 파일/헝크만 `git add <경로>` 또는 `git add -p`로
   스테이징한다. 관련 없는 변경이 함께 들어가지 않게 `git diff --staged`로 재확인.
4. **메시지 작성(한글).** 아래 형식으로 작성한다.
5. **커밋.** `git commit`. 다음 단위가 남아 있으면 2~5를 반복한다.
6. **보고.** 만든 커밋들을 `git log --oneline -n <개수>`로 보여주고 요약한다.

> 푸시는 사용자가 명시적으로 요청할 때만 한다. 현재 브랜치가 기본 브랜치(main/master)
> 라면, 커밋 전에 작업 브랜치를 만들지 사용자에게 확인한다.

## 메시지 형식

```
<type>(<scope>): <요약 (한글, 50자 내외)>
<빈 줄>
<본문: 무엇을, 왜 바꿨는지 (한글).bullet point로 줄 구분하여 작성>
<빈 줄>
<푸터: BREAKING CHANGE / Closes #이슈 등>
```

### type (필수)

| type | 쓰임 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서만 변경 |
| `style` | 동작에 영향 없는 포맷/세미콜론/공백 등 |
| `refactor` | 기능 변화 없는 코드 구조 개선 |
| `perf` | 성능 개선 |
| `test` | 테스트 추가/수정 |
| `build` | 빌드 시스템·의존성 변경 |
| `ci` | CI 설정/스크립트 변경 |
| `chore` | 그 외 잡무(빌드·소스 외) |
| `revert` | 이전 커밋 되돌리기 |

### scope (선택)
변경이 영향을 준 영역. 모듈·기능·패키지명 등. 예: `feat(auth):`, `fix(payment):`.
범위가 넓거나 애매하면 생략한다.

### 요약 줄 규칙
- 한글로, 무엇을 했는지 간결하게. 50자 내외.
- 끝에 마침표를 찍지 않는다.
- "무엇을 추가/수정했다"가 한눈에 보이게(명사형 `~추가`/`~수정`도 좋다).

### 본문 규칙
- **무엇을** 바꿨고 **왜** 바꿨는지를 쓴다(코드를 보면 아는 "어떻게"보다 "왜"가 중요).
- bullet point로 줄 구분하여 가독성 좋게 작성
- 변경이 자명한 작은 커밋(오타·포맷 등)은 본문 생략 가능.


### 푸터 규칙
- 호환성 깨짐: `BREAKING CHANGE: <설명>`
- 이슈 연결: `Closes #123`, `Refs #123`

## 기능 단위로 나누는 기준

- 서로 다른 type이면 나눈다(예: 기능 추가 + 리팩터링 → `feat` 커밋, `refactor` 커밋).
- "이 커밋만 되돌려도 말이 되는가?"가 단위 판단 기준이다.
- 한 줄 요약에 "그리고/및"이 들어가면 보통 두 커밋으로 나눠야 한다는 신호다.
- 대규모 변경이라도 의미 단위로 쪼개서 리뷰·되돌리기가 쉽게 한다.

## 예시

```
feat(search): 상품명 부분 일치 검색 기능 및 GIN 인덱스 추가

- 기존 : 정확히 일치하는 상품명만 조회 
- 변경 : LIKE 기반 부분 일치 검색 도입, name 컬럼에 GIN 인덱스 추가
- 기대효과 : 일부만 입력해도 검색 가능, 풀스캔 방지

Closes #142
```

를 기본으로 넣지 않는다. 트레일러를 넣고 싶으면 사용자가 명시적으로 요청할 때만 추가한다.

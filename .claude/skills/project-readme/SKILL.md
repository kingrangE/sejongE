# project-readme

전체 Plan이 끝났을 때, 프로젝트의 README를 "잘 설계하고 관리한 개발자가 직접 쓴 것처럼"
정리하는 스킬이다. `portfolio-doc`가 단계별 깊은 의사결정 기록이라면, 이 스킬은 그것을
재료로 **프로젝트 전체를 한눈에 설득**하는 최상위 문서를 만든다.

## 절대 규칙 — 읽는 사람이 누구인가

독자는 **기술 면접관/시니어 리뷰어**다. 그들은 과장과 AI 생성물을 즉시 알아챈다.
모든 문장은 실제 코드/커밋/설정 파일에서 확인되는 사실이어야 한다. 확인되지 않는 건
쓰지 않는다. "있어 보이는 말"보다 "구체적인 결정과 트레이드오프"가 신뢰를 만든다.

## 작성 절차

1. **근거 수집(추측 금지).**
   - `docs/portfolio/` 의 단계 문서들을 읽는다 — 이미 정리된 의사결정의 1차 근거.
   - 실제 스택을 코드에서 확인한다: `package.json`/`requirements.txt`/`go.mod`/
     `pom.xml` 등 의존성, 디렉터리 구조, 설정 파일(Docker, CI, env), 진입점.
   - `git log --oneline`(가능하면)으로 개발 흐름과 단계를 파악한다.
2. **기존 README 보존.** README.md가 이미 있으면 먼저 읽는다. 사람이 쓴 실제 정보
   (설치법, 라이선스, 배지, 스크린샷 경로 등)는 지우지 말고 다듬어 살린다. 전면
   삭제 후 새로 쓰지 않는다.
3. **구조에 맞춰 작성.** `references/readme-structure.md`의 골격을 따르되, 프로젝트에
   해당 없는 섹션은 넣지 않는다(빈 섹션을 채우려고 내용을 지어내지 말 것).
4. **AI 티 점검.** 초안을 `references/ai-tells-checklist.md`로 자가 점검하고 고친다.
5. **저장 + 보고.** 루트 `README.md`에 쓰고, 무엇을 근거로 어떤 섹션을 채웠는지,
   확인 못 해 비워 둔 부분이 있는지 1~2줄로 사용자에게 알린다.

## 무엇이 반드시 보여야 하는가

- **기술 스택(실물):** "최신 기술 활용" 같은 문구 금지. `React 18 + TypeScript`,
  `PostgreSQL 15`, `Redis 캐시 레이어`처럼 버전·역할까지 구체적으로. 각 기술이 *왜*
  거기 있는지 한 줄 근거를 붙인다.
- **설계 의사결정:** "X를 고려했고 → Y 제약 때문에 → Z를 택했다 → 그래서 이렇게
  구현했다"는 인과가 드러나야 한다. 핵심 결정 2~4개를 골라 짧게 서술하거나, 깊은
  내용은 `docs/portfolio/`로 링크한다.
- **개발 단계(실무적):** 기능 나열이 아니라, 어떤 순서로 무엇을 먼저 세우고
  (예: 데이터 모델 → 인증 → 핵심 도메인 → API → 프론트), 단계마다 무엇을 검증했는지.
- **솔직한 한계/다음 단계:** 트레이드오프와 미완 영역을 한 단락. 이게 오히려 성숙도를
  보여준다.

## 작성 원칙(문체)

- 한국어, 담백한 실무 문체. 1인칭 과시("저는 완벽하게…")도, 마케팅 톤("혁신적인…")도
  쓰지 않는다.
- 사실 → 이유 → 결과 순의 짧은 문장. 모든 섹션을 대칭으로 채우려 하지 말고, 할 말이
  많은 곳은 길게, 없는 곳은 짧게(자연스러운 비대칭).
- 코드에서 검증 안 되는 수치·성과를 만들어내지 않는다. 측정값이 없으면 정성적으로 쓴다.

# portfolio-doc

구현 결과를 "포트폴리오에 쓸 수 있는 기술 설명 문서"로 자동 정리하는 스킬이다.
단순 변경 로그가 아니라, **왜 그렇게 만들었는지(의사결정의 근거)**를 독자가 이해할 수
있게 쓰는 것이 목표다.

## 언제 실행되나

이 스킬은 두 경로로 실행된다.

1. **수동** — 사용자가 특정 기능/작업을 포트폴리오로 정리해 달라고 요청할 때.
2. **자동(하이브리드)** — `.claude/hooks/portfolio_on_todo.py` hook이 TodoWrite를
   감시하다가, Plan의 한 단계가 `completed`로 바뀌는 순간 `[portfolio-doc 자동 트리거]`
   라는 컨텍스트를 주입한다. 그 컨텍스트가 보이면, **다음 작업을 진행하기 전에** 방금
   완료된 단계에 대해 이 스킬을 한 번 실행한다.

> 예: Plan에 5개 단계가 있으면, 1단계 완료 → 1개 문서, 2단계 완료 → 1개 문서 … 처럼
> 단계마다 한 번씩 호출된다. 단계 수는 5개가 아니어도 동일하게 동작한다.

## 작성 절차

각 단계를 문서화할 때 다음 순서로 진행한다.

1. **무엇이 완료됐는지 파악.** 자동 트리거면 주입된 컨텍스트의 단계 설명을 사용한다.
   수동이면 사용자가 지목한 기능/범위를 사용한다.
2. **근거 수집.** 해당 단계에서 바뀐/추가된 코드를 읽는다(`git diff`, 관련 파일).
   추측하지 말고 실제 코드·커밋·대화 맥락에서 의사결정의 근거를 모은다. 근거가
   부족한 항목(특히 "대안")은 비워 두기보다 "확인 필요"로 표시하고 사용자에게 묻는다.
3. **문서 작성.** `references/section-template.md`의 섹션 구조를 그대로 채운다.
   - 적용 이유 · 설계 고민 · 대안과 비교(장단점 표) · 기술 설명은 필수 섹션이다.
   - "기술 설명"은 그 기술을 처음 보는 사람도 이해하도록 *무엇이고 어떻게 동작하는지*
     를 먼저 쓰고, 그 다음 *이 프로젝트에서 어떻게 쓰였는지*를 잇는다.
   - 장단점은 반드시 **선택한 안 vs 대안**을 같은 기준으로 비교하는 표로 만든다.
4. **파일 저장(단계 파일 + 인덱스, 둘 다).**
   - 단계 파일: `docs/portfolio/NN-<짧은-기능명>.md` (NN은 완료 순서 2자리, 예 `01`).
     같은 단계를 다시 문서화하면 새 파일을 만들지 말고 기존 파일을 갱신한다.
   - 인덱스: `docs/portfolio/README.md`에 한 줄 추가/갱신
     (`- [NN 제목](NN-파일.md) — 한 줄 요약`). 없으면 새로 만든다.
5. **간단히 보고.** 어떤 파일을 썼고 무엇을 채웠는지, "확인 필요"로 남긴 항목이
   있는지 사용자에게 1~2줄로 알린다. 그 뒤 원래 진행하던 다음 단계로 돌아간다.

## 작성 원칙

- **한국어**로, 면접관/리뷰어가 읽는다고 가정하고 쓴다. 과장 없이 트레이드오프를
  솔직하게 적는 편이 더 설득력 있다.
- 자동 트리거로 호출됐을 때, 문서화는 한 단계분만 하고 끝낸다. 전체 프로젝트를 한꺼번에
  다시 쓰지 않는다(중복·맥락 손실 방지).
- 코드에서 확인되지 않는 사실을 지어내지 않는다. 모르면 "확인 필요"로 남긴다.
- 번호(NN)는 "완료된 순서"다. 인덱스의 마지막 번호 다음을 쓰면 된다.
